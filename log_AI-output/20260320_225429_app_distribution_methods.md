# ユーザーの入力
@[main.py]と動作に必要なファイルやライブラリをアプリとして他者に配布したいと考えている。
適切な方法を調査し、説明を行え。

# AIの出力
Pythonで作成したアプリケーション（`main.py`）と必要な環境を、Pythonがインストールされていない他者のPCでも動作する「アプリ（実行可能ファイル）」として配布するには、プログラムを **`.exe` ファイルに変換（パッケージ化）** するのが最も一般的かつ適切な方法です。

現在、アプリの動作に必要な要素は以下の通りです。
- **プログラム本体**: `main.py`
- **データファイル**: `PDM_ARF-001.docx` (コード内で読み込んで操作しています)
- **外部ライブラリ**: `customtkinter`, `python-docx` (`docx` としてインポートされています)

これらを一つにまとめて配布するための具体的なツールと手順をご説明します。

---

### おすすめのツール：「PyInstaller」または「auto-py-to-exe」

Pythonプログラムを `.exe` に変換するデファクトスタンダードは **PyInstaller（パイインストーラー）** です。
また、これを視覚的な操作画面（GUI）で簡単に使えるようにした **auto-py-to-exe** というツールもあります。今回はコマンド操作で完結するPyInstallerを基準に説明しますが、コマンド操作に抵抗がある場合は `auto-py-to-exe` を使うのもおすすめです（内部では同じPyInstallerが動いています）。

### 配布に向けた2つのアプローチ
PyInstallerで変換・ビルドする際、大きく分けて2つの形式を選択できます。

**① 1つのフォルダにまとめる方法（おすすめ）**
EXEファイルと、動作に必要な関連ライブラリがすべて入った1つのフォルダが生成されます。そのフォルダの中に `PDM_ARF-001.docx` を手動でコピー追加し、フォルダごとZIP等で圧縮して配布します。
- **メリット**: `main.py` のコードを書き換える必要がないため、最もエラーになりにくく確実です。
- **デメリット**: フォルダ内にライブラリなどのファイルが多数並ぶため少しごちゃごちゃします。

**② 1つのファイル（.exe）に完全にまとめる方法 (Onefile)**
すべてが内包された1つの `.exe` だけで完結します。
- **メリット**: 見た目がすっきりしており、1ファイルを渡すだけなので利用者が分かりやすいです。
- **デメリット**: 実行時に裏で一時フォルダに内部データが解凍される仕様上、`main.py` のファイルの読み込み処理（10行目のパス取得部分）を少し書き換える必要があります。また、毎回解凍が行われるため起動に少し時間がかかります。

---

### アプリ化の手順（① 1つのフォルダにまとめる場合）

現在のコード構造を変更せずにそのまま配布できる「①」の手順を解説します。

#### 1. PyInstaller のインストール
VSCodeなどのターミナルを開き、以下のコマンドを実行してインストールします。
```bash
pip install pyinstaller
```

#### 2. .exe ファイルの生成（ビルド）
ターミナルで `main.py` があるディレクトリ（`PDE-ARF`）にいることを確認し、以下のコマンドを実行します。
```bash
pyinstaller --noconsole main.py
```
- `--noconsole` オプションをつけることで、アプリ実行時に黒いコマンドプロンプト画面（コンソール）が表示されなくなり、ちゃんとした専用アプリのようになります。

#### 3. 必要なファイルの配置
ビルドが完了すると、新しく `dist` というフォルダが作成され、その中に `main` というフォルダが生成されます。
この `main` フォルダの中に `main.exe` という実行ファイルがあります。

現在の動作仕様上、`main.exe` と同じ階層に `PDM_ARF-001.docx` が必要ですので、**`dist/main/` フォルダの中に、元の `PDM_ARF-001.docx` をコピーして配置**してください。

#### 4. 配布
`PDM_ARF-001.docx` を入れた状態の `main` フォルダの名前を、分かりやすいアプリ名（例：`会計要望書生成アプリ`）に変更し、フォルダごとZIPファイルに圧縮します。
このZIPファイルを相手に渡し、解凍して中にある `.exe` をダブルクリックしてもらえれば、Python環境がないPCでもアプリが起動します。

---

### （補足）1つのファイル（.exe）に完全にまとめたい場合

もし②の方法で完全な1ファイルのアプリを目指す場合は、`main.py` の `base_dir` の定義部分（10行目付近）を以下のように書き換える必要があります。

**変更前:**
```python
import os

base_dir = os.path.dirname(os.path.abspath(__file__))
```

**変更後:**
```python
import os
import sys

if hasattr(sys, '_MEIPASS'):
    # PyInstallerで1ファイル化された場合に解凍される一時フォルダのパス
    base_dir = sys._MEIPASS
else:
    # 通常のPythonスクリプトとして実行した場合のパス
    base_dir = os.path.dirname(os.path.abspath(__file__))
```

このコード変更を行ったうえで、以下のコマンドでビルドします。
```bash
pyinstaller --noconsole --onefile --add-data "PDM_ARF-001.docx;." main.py
```
※これにより `PDM_ARF-001.docx` も内包された完全な一つの `main.exe` が出来上がります。

まずは簡単で失敗しにくい「**① 1つのフォルダにまとめる方法**」から試してみることをお勧めします。

# ユーザーの入力
base_dirについて、この処理は絶対パスを規定するための処理だと考えられるが、これを相対パスの処理とした場合は、この変更は不要にはならないか？

# AIの出力
結論から申し上げますと、**相対パスに変更してもあの分岐処理は不要にはなりません。むしろ、エラーの原因になるため分岐処理が必須となります。**

その理由は、PyInstallerの `--onefile`（1ファイル化）特有の動作仕様によるものです。

### 【理由1】データファイルは裏で「一時フォルダ」に解凍される
PyInstallerで `--onefile` を指定してデータを `.exe` の中に埋め込んだ場合、実行時にそのデータファイル（今回なら `PDM_ARF-001.docx`）は、ユーザーには見えないPCの裏側の **「一時フォルダ (Tempフォルダ)」 にこっそり解凍**して配置されます。

### 【理由2】プログラムの「現在地（カレントディレクトリ）」は元のまま
しかし、アプリが実行されている間、プログラム自身が「自分が今どこにいるか（カレントディレクトリ）」と認識している場所は、**ユーザーが `.exe` をダブルクリックした元のフォルダ**のままです。

もし `__file__` による絶対パス取得をやめて、単なる相対パスで `docx.Document("PDM_ARF-001.docx")` とだけ書いたとします。
すると、プログラムは「なるほど、自分が今いる場所（`.exe` があるフォルダ）の中にあるんだな」と思ってそこを探してしまいます。しかし、実際のファイルは「一時フォルダ」に解凍されているため、「ファイルが見つからない」というエラーでアプリが落ちてしまいます。

だからこそ、「自分が今実行されている一時フォルダの絶対パス（`sys._MEIPASS`）」を明示的に取得し、確実に一時フォルダ内にあるファイルを読みに行かせる処理がどうしても必要になります。

---

### （補足）相対パスで動くケース
「データファイルを `.exe` の中に埋め込まない」という選択をした場合、つまり `.exe` と並べて同じフォルダに `PDM_ARF-001.docx` を手動で置くことにすれば、相対パスでも動きます。
（これは前回ご説明した「① 1つのフォルダにまとめる方法」でフォルダごと配布する場合と同じ運用です）

しかし、これだと結局ファイルが2つになってしまうため、無理に `--onefile` で1ファイル化するメリットが大半なくなってしまいます。

このような理由から、**1ファイルの中にデータごと完全に封入して配布したい場合に限り、あの絶対パス取得の分岐が必要**になります。
